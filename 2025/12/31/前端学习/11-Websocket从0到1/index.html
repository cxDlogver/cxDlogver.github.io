WebSocket

1. WebSocket的诞生 —— HTTP的局限性

【HTTP的局限性】

HTTP 协议是 请求-响应模型，它的核心特征可以用一句话概括：

客户端先发请求，服务器才能响应

也就是说：

  • 服务器 不能主动 给客户端发消息
  • 一次请求，对应一次响应
  • 通信是短连接或逻辑上的短会话

在 Web 应用的发展过程中，“实时性”几乎是一个绕不开的话题。聊天室、在线协作、即时通知、游戏状态同步，这些场景都有一个共同点：服务器需要在“任何时刻”主动把消息推送给客户端。

为了“模拟”实时通信，开发者在 HTTP 之上发明了几种方案。

轮询（Polling）

客户端每隔一段时间就问一次服务器：

“有新消息吗？”

问题：

  • 大量无意义请求
  • 延迟取决于轮询间隔
  • 浪费服务器和网络资源
长轮询（Long Polling）

客户端发起请求后，服务器先不立即返回，等有数据再返回。

然后客户端立刻再发下一次请求。

比轮询好，但仍然存在问题：

  • 本质仍是 HTTP 请求
  • 高并发下服务器压力大

Server-Sent Events（SSE）

服务器可以单向推送数据给客户端。

限制很明显：

  • 只能服务端 → 客户端
  • 客户端不能实时向服务端发送消息
  • 聊天室、协作编辑等场景不适用

无论哪种方案，本质问题始终存在：

HTTP 并不是一个为“持续、双向通信”设计的协议

我们真正需要的是：

  • 一条 长期存在的连接
  • 客户端和服务器 地位对等
  • 任意一方都能 随时发送消息
  • 低延迟、低开销

这正是 WebSocket 出现的背景。

【WebSocket的诞生】

WebSocket 并不是 HTTP 的“改良版”，而是一个全新的通信协议，它的目标非常明确：

在浏览器与服务器之间建立一条真正的、持久的、全双工通信通道

关键词拆解一下：

关键词 含义
持久 连接一旦建立，不用反复创建
全双工 客户端和服务器可同时发送
实时 消息可以立即到达
低开销 没有重复的 HTTP 头部
WebSocket 与 HTTP 的关系

一个容易误解的点是：

WebSocket 不是 HTTP 的替代品

它们的关系是：

  • 建立连接时使用 HTTP
  • 建立完成后不再使用 HTTP

具体来说：

  1. 浏览器发起一个 HTTP 请求
  2. 请求中声明：我要升级为 WebSocket
  3. 服务器同意升级
  4. 协议从 HTTP 切换为 WebSocket
  5. 后续通信全部走 WebSocket 帧

你可以理解为：

HTTP 负责“牵线搭桥”，WebSocket 负责“长期交流”

WebSocket 的核心特性
  1. 真正的双向通信（Full-Duplex）

在 WebSocket 中：

  • 客户端 可以随时发消息
  • 服务器 也可以随时推送
  • 不需要“请求—响应”这一前提

这使得聊天、推送、协作变得非常自然。

  1. 单连接、长期存在

WebSocket 连接一旦建立：

  • 不会频繁创建和销毁
  • 不需要反复携带 HTTP Header
  • 网络和 CPU 开销显著降低

对于高并发场景尤其重要。

  1. 更轻量的消息格式

HTTP 每次请求都包含大量头部信息，而 WebSocket 使用的是二进制帧（Frame）：

  • 帧结构简单
  • 传输效率高
  • 基于 WebSocket + JWT 鉴权，结合自定义心跳与指数退避重连，解决实时链路假死、断连与权限失效问题，实现多源传感器数据稳定低延迟推送。
  1. 原生浏览器支持

现代浏览器内置了 WebSocket API：

无需插件、无需额外协议栈，这使 WebSocket 成为 Web 实时通信的事实标准。

3. WebSocket 完整生命周期

WebSocket 是一种基于 TCP 的全双工通信协议。与传统 HTTP 不同，WebSocket 建立的是长连接，其复杂度并不在“连接本身”，而在连接建立之后的生命周期管理。

在实现一个WebSocket功能是，需要考虑 WebSocket 从 建立连接 → 鉴权 → 状态绑定 → 心跳 → 业务处理 → 异常与清理 的完整生命周期。

【连接建立阶段（Connection Establishment）】

1. 基于 HTTP 的 Upgrade 握手

WebSocket 的连接并不是一开始就使用 WebSocket 协议，而是：

  1. 客户端先发起 HTTP 请求
  2. 通过 Upgrade: websocket 头部请求协议升级
  3. 服务器返回 101 Switching Protocols
  4. 连接从 HTTP 升级为 WebSocket

此阶段的本质是：

WebSocket 是通过 HTTP 建立连接，但不会继续使用 HTTP 进行通信

一旦握手完成：

  • HTTP 协议结束
  • 后续通信完全基于 WebSocket 帧协议
2. 连接成功的误区

需要特别强调：

WebSocket 连接成功（open） ≠ 用户身份合法 ≠ 业务可用

open 事件只说明：

  • 网络通信正常
  • 协议升级成功

并不代表：

  • 用户已登录
  • 权限已校验

【实时通信阶段（WebSocket Event Model）】

连接建立后，通信通过 WebSocket 的事件模型进行。

1. 核心事件
  • open：连接建立成功
  • message：接收对端消息
  • error：底层通信异常
  • close：连接关闭
2. 数据收发模型
  • 通过 send() 发送数据
  • 所有业务消息通过 message 接收
  • WebSocket 本身不理解业务数据结构

因此，必须在应用层自行定义消息协议。

【连接级鉴权（Authentication）】

1. 为什么 WebSocket 必须单独鉴权

WebSocket 与 HTTP API 最大的不同在于：

  • 没有 request body
  • 没有 REST 路由
  • 一次连接，长期存在

如果不进行连接级鉴权，会导致：

  • 任意用户可建立长连接占用资源
  • 可伪造身份发送业务消息
  • 权限控制完全失效
2. 常见鉴权方式

工程中最常见、最可靠的方式是：

  • Token 鉴权（JWT / Access Token）
  • 首包鉴权（open 后第一条消息）

典型流程：

  1. HTTP 登录获取 token
  2. WebSocket 建立连接
  3. 客户端发送 auth 消息
  4. 服务端校验 token
  5. 鉴权成功后才允许业务通信

这在语义上等价于 HTTP 中的 Authorization。

3. 鉴权状态机

在 AUTHED 之前：

  • 不允许处理任何业务消息
  • 非法消息直接关闭连接

【权限控制（Authorization）】

1. 鉴权 ≠ 授权

这是非常容易混淆的一点：

  • 鉴权（Authentication）：你是谁
  • 授权（Authorization）：你能做什么

例如在聊天室系统中：

  • 普通用户：发消息、进群
  • 管理员：踢人、禁言、解散群聊
2. 权限数据来源

权限信息通常存储在：

  • 数据库
  • Redis
  • 内存缓存

【心跳机制（Heartbeat）】

1. 为什么必须有心跳

WebSocket 的连接在以下情况下不会立即触发 close：

  • 客户端断网
  • NAT / 代理回收连接
  • 移动网络切换

没有心跳会产生大量“假在线”连接。

  • WebSocket 原生支持
  • 不携带业务信息

  • 定期确认连接是否存活
  • 超时即主动关闭

【异常处理与重连策略（Error & Reconnect）】

1. 异常来源
  • 网络中断
  • 心跳超时
  • 鉴权失败
  • 协议错误
2. 客户端重连

WebSocket 不会自动重连，需要客户端自行实现：

  • 指数退避
  • 最大重试次数
  • 网络恢复触发

【连接关闭与资源清理（Cleanup）】

1. 关闭来源
  • 客户端主动关闭
  • 服务端踢下线
  • 心跳失败
  • 异常断开
2. 必须做的清理工作
  • 从 userId -> ws 映射中移除
  • 释放内存状态
  • 停止心跳定时器

所有资源清理必须放在 close 阶段完成。

【总结】

WebSocket 并不是“连上就完事”的协议，而是一套贯穿整个连接生命周期的状态管理体系。

真正决定系统稳定性与安全性的，从来不是 new WebSocket()，而是：

鉴权是否严格、状态是否清晰、心跳是否可靠、关闭是否彻底。

4. WebSocket 鉴权

可以将 WebSocket 鉴权抽象为三个核心问题：

1. 身份确认（Authentication）

  • 连接的“这个人”是谁？
  • 是否是合法用户？
  • 是否被伪造？
  1. 权限控制（Authorization）
  • 这个连接能做什么？
  • 能否加入某个房间 / 群聊 / 频道？
  • 是否有发送、接收、管理权限？
  1. 会话可信性（Session Trust）
  • 连接是否被劫持？
  • Token 是否过期？
  • 是否支持强制下线？

所有 WebSocket 鉴权方案，本质都遵循：

在连接生命周期内，某个时刻完成一次可信的身份确认。

可用的入口只有三类：

  1. HTTP 握手阶段
  2. 连接建立后的第一条消息
  3. 连接生命周期内的持续校验

【HTTP 握手阶段鉴权】

1. 原理

WebSocket 建立时，本质是一次 HTTP 请求：

此时：

  • 浏览器仍然会发送 Cookie
  • 可以附加 URL Query
  • 可以在 Node / Java / Go 服务端拦截
2. 常见携带方式

（1）Cookie 鉴权（最自然）

  • 前提：
    • 已通过 HTTP 登录
    • Session / Token 已写入 Cookie
  • 握手时浏览器自动携带

优点：

  • 前后端无感
  • 与传统登录系统完全兼容

风险点：

  • CSRF 风险

（2）URL Query Token

服务端在 upgrade 阶段解析：

优点：

  • 实现简单
  • 适合移动端 / 非浏览器客户端

缺点：

  • Token 可能出现在日志中
  • 不适合长期敏感凭证

（3）自定义 Header（非浏览器）

限制：

  • 浏览器 不允许
  • Node / 原生客户端可用
3. 服务端典型流程
  1. upgrade 事件触发
  2. 提取 Token / Session
  3. 校验合法性
  4. 失败 → socket.destroy()
  5. 成功 → wss.handleUpgrade()

【连接建立后首条消息鉴权】

1. 原理
  • WebSocket 建立成功
  • 不立即信任
  • 要求客户端发送一条 AUTH 消息
2. 服务端逻辑
  • 初始连接状态：unauthenticated
  • 收到第一条消息：
    • 校验 token
    • 标记该 socket 为已登录用户
  • 未通过鉴权：
    • 立即 close
3. 优缺点分析

优点：

  • 不依赖 HTTP 握手
  • 适合复杂自定义协议

缺点：

  • 连接已经建立
  • 需要额外状态管理
  • 容易被滥连攻击（DoS）

工程建议：仅在无法控制握手阶段时使用

【Token（JWT / Session）】

在 WebSocket 系统中，Token 鉴权是最主流、也是最成熟的方案。根据 Token 是否“自包含状态”，工程上通常分为 JWT（无状态）模式 与 Session / Redis（有状态）模式。两者在设计理念、适用场景与系统代价上存在本质差异。

1 JWT 模式（无状态鉴权）
  1. 核心设计思想

JWT（JSON Web Token）是一种自包含 Token，其载荷中直接包含身份与有效性信息，服务端不需要保存会话状态。

一个典型 JWT 中包含：

  • userId：用户唯一标识
  • exp：过期时间
  • iat：签发时间
  • iss / aud：签发方 / 受众
  • signature：签名，用于防篡改

服务端只需：

  • 校验签名
  • 校验时间有效性
  • 解析用户信息

即可完成身份认证。

  1. 工程特性
  • 服务端无状态
  • 不依赖数据库 / Redis
  • 任意节点均可校验
  • 非常适合横向扩展

这使得 JWT 成为 WebSocket 场景中最常见的鉴权方案之一。

  1. 适用场景

JWT 模式尤其适合：

  • 分布式 WebSocket 服务
  • 多实例 / 多节点部署
  • WebSocket Gateway 架构
  • 云原生、容器化部署
  • Serverless / 边缘节点

在这些场景下：

“连接落在哪个节点”是不确定的，JWT 的无状态特性可显著降低系统复杂度

  1. WebSocket 中的关键问题：Token 过期

与 HTTP 不同，WebSocket 是长连接，这引出一个必须面对的问题：

连接存续期间，JWT 可能会过期

工程上必须提前明确策略：

策略一：Token 过期后允许连接继续
  • 优点：实现简单、连接稳定
  • 风险：安全窗口扩大
  • 适合：低风险业务、短连接周期
策略二：Token 过期后强制断连
  • 优点：安全性高
  • 代价：用户体验下降
  • 实现方式：定时检查 / 心跳时校验
策略三：Token 刷新机制（推荐）
  • 使用短期 Access Token
  • 配合 Refresh Token
  • 在连接内完成刷新
  • 刷新失败则断连

关键结论：

JWT 模式在 WebSocket 中，必须显式设计“过期处理策略”，不能沿用 HTTP 的隐式假设。

  1. JWT 模式的优缺点总结

优势：

  • 无状态，扩展性极佳
  • 实现简单
  • 适合高并发、大规模连接

劣势：

  • 无法主动失效
  • 过期控制复杂
  • 权限变更不即时生效（可以手动添加处理）
2. Session / Redis 模式（有状态鉴权）
  1. 核心设计思想

Session 模式的本质是：

Token 只是一个索引，真正的身份状态保存在服务端

流程如下：

  1. 客户端携带 Token
  2. 服务端通过 Token 查询 Redis / 内存
  3. 获取 Session 数据（userId / roles / permissions）
  4. 判定身份与权限
  1. 工程特性
  • 服务端有状态
  • 依赖 Redis / 内存存储
  • 每次鉴权都需要一次存储访问
  • 可以随时修改 / 删除 Session
  1. 适用场景

Session 模式非常适合以下场景：

  • 强权限控制系统
  • 管理后台 / 运维平台
  • 需要“强制下线”的系统
  • 金融、风控、后台管理类业务

在这些系统中：

“能否立刻失效” 比 “是否无状态” 更重要

  1. WebSocket 中的优势

在 WebSocket 场景下，Session 模式具备显著优势：

  • 可实时踢下线
  • 权限变更立即生效
  • 可绑定设备 / IP / 登录来源
  • 可限制同时在线数量

例如：

  • 管理员修改权限后，连接可立刻失效
  • 发现异常行为，立即断开连接

5.WebSocket 心跳机制

【心跳机制】

WebSocket 是长连接，以下情况不会立即触发 close 事件：

  • 客户端突然断网
  • 手机锁屏 / 切后台
  • NAT / 代理回收空闲连接
  • 路由器重启

结果是：

  • 服务端以为连接还在
  • 客户端也可能以为连接还在
  • 出现大量“假在线”连接

没有心跳的后果

  • 在线人数统计不准确
  • 消息发送失败但无感知
  • 连接池被“僵尸连接”占满
  • 无法及时踢下线、重连

结论：

在任何生产环境中，WebSocket 必须实现心跳机制。

心跳机制的核心目标

  1. 检测连接是否仍然可达
  2. 尽早发现断线
  3. 触发重连或资源清理
  4. 保证在线状态的真实性

【两种主流方式】

协议级 Ping / Pong（WebSocket 原生）

这是 WebSocket 协议（RFC 6455）自带的能力。

特点

  • 不进入业务层
  • 不暴露给前端 API（浏览器端无法直接发送 ping）
  • Node.js（如 ws）支持

工作方式

  • 服务端定期发送 ping 帧
  • 客户端自动回复 pong
  • 若长时间未收到 pong，判定连接失效

优点

  • 轻量
  • 不干扰业务协议
  • 标准化

缺点

  • 浏览器端无法主动发送 ping
  • 不适合承载业务级状态（如 token 校验）

server.js

重要说明

  • 当服务端 ping时：
    • 客户端会自动回复 pong
    • 不会触发 pong 事件
  • pong 事件只在你主动 ws.ping() 时才触发

浏览器 JS 层：

  • ❌ 看不到 ping
  • ❌ 看不到 pong
  • ❌ 不触发 message
  • ❌ 不触发任何“心跳相关事件”

【应用层自定义心跳】

通过普通 WebSocket 消息实现心跳。更为常用。

特点

  • 完全由业务层控制
  • 浏览器、Node 统一
  • 可结合鉴权、权限、状态校验

优点

  • 灵活
  • 可扩展
  • 适合 WebSocket 业务系统

缺点

  • 需要自己实现逻辑
  • 稍有额外流量

  • 在 close / error 时触发重连

【区分正常close和异常close】

根据判断正常close和异常close来决定是否处理或者重传。

看 close code

  • 1000：正常关闭
  • 1006：异常关闭（端口/网络断）

看 error 事件

  • 异常断开：常伴随 error
1）正常关闭（Graceful Close）

定义：按照 WebSocket 协议完成 Close 握手。

常见场景

  • 客户端主动退出
  • 服务端业务结束
  • 鉴权失败后主动 close()
  • 被“踢下线”（策略性关闭）

特征

  • onclose 一定触发
  • code 为 1000 或明确的策略码
  • 双方都“知道”这次关闭
2）异常关闭（Abnormal Close / 端口断）

定义：没有完成 Close 握手，连接直接消失。

常见场景

  • 客户端断网 / 进程被杀
  • 服务端进程崩溃
  • NAT / 代理回收端口
  • TCP 连接被强制重置

特征

  • onclose 触发
  • code 通常为 1006
  • reason通常未空
  • 往往伴随 error
code 含义 是否正常
1000 Normal Closure ✅ 正常
1001 Going Away ✅ 正常
1008 Policy Violation ✅ 正常（策略关闭）
1011 Internal Error ⚠️ 服务端异常
1006 Abnormal Closure ❌ 异常（端口/网络）

ws.close() 支持传递 code 和 reason，并且这是工程上强烈推荐的做法。

下面把用法、合法范围、注意事项、与浏览器差异一次讲清楚。

最直接的结论

  • code：WebSocket Close Code（必需是合法值）
  • reason：关闭原因（可选，字符串）
  1. 正常关闭
  1. 鉴权失败 / 被踢下线（常用）
  1. 服务端异常

官方close code

code 含义
1000 正常关闭
1001 端点离开（刷新/切页）
1006 异常断开（浏览器推断，不能手动设置）
1008 策略违规（如权限问题，但不够细）
1011 服务端内部错误
