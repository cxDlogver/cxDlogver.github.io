2. 什么是 RESTful（口述回答模板）

RESTful 是一种 接口设计风格，核心思想是把服务器资源当成资源来设计，通过 URL 表示资源，通过 HTTP 方法表示操作。

在 RESTful 规范中，URL 通常使用名词表示资源，比如用户、订单；
具体做什么操作由 HTTP 方法决定，比如 GET 用来查询，POST 用来新增，PUT 或 PATCH 用来修改，DELETE 用来删除。

RESTful 强调接口语义清晰、风格统一、前后端解耦，因此在前后端分离项目中使用非常普遍。

Promise

面试题一：什么是 Promise？为什么要引入 Promise？

口述回答模板：

Promise 是 JavaScript 中用于处理异步操作的一套标准化解决方案，本质上它是一个对象，用来描述“未来某一时刻才会有结果”的操作。

在 Promise 出现之前，JavaScript 的异步逻辑主要依赖回调函数实现。但当异步流程变复杂之后，会出现两个明显问题：
第一是回调地狱，多个异步操作层层嵌套，代码结构非常混乱，可读性和可维护性都很差；
第二是错误处理分散，每一层回调都需要单独处理异常，很难做到统一管理。

Promise 的核心价值在于：把异步操作本身抽象成一个对象，通过状态管理和链式调用来组织异步流程。这样可以用更接近同步的写法描述异步逻辑，同时集中处理成功和失败情况。

在工程实践中，Promise 不仅解决了回调地狱和错误分散的问题，也为 async / await 的出现提供了基础，使异步代码的可读性和可维护性显著提升。

面试题二：Promise 有哪几种状态？状态是如何流转的？

口述回答模板：

Promise 在整个生命周期中只有三种状态：pending、fulfilled 和 rejected。
pending 表示初始状态，说明异步操作正在进行但尚未完成；
fulfilled 表示异步操作成功完成；
rejected 表示异步操作执行失败。

Promise 的状态流转有两个非常重要的特性。
第一，状态不可逆，只能从 pending 变为 fulfilled 或 rejected，不能反向切换；
第二，状态只会改变一次，无论 resolve 还是 reject，只要其中一个生效，后续调用都会被忽略。

这种设计的意义在于保证异步结果的确定性。只要 Promise 的状态已经确定，无论在什么时候通过 then 或 catch 去订阅，都一定能拿到相同的结果，这也是 Promise 可以被安全复用和组合的基础。

面试题三：Promise 是如何实现链式调用的？为什么可以一直 then 下去？

口述回答模板：

Promise 能实现链式调用的根本原因是：then 方法本身一定会返回一个新的 Promise 对象。

当上一个 Promise 状态变为 fulfilled 或 rejected 时，then 中对应的回调函数会被放入微任务队列执行。这个回调函数的返回值，会直接决定下一个 Promise 的状态。

如果回调返回的是普通值，这个值会被自动包装成一个 fulfilled 状态的 Promise；
如果返回的是一个 Promise，那么下一个 Promise 会直接“接管”它的状态；
如果在回调中抛出异常，下一个 Promise 就会进入 rejected 状态。

正是因为这种“返回新 Promise + 状态接力”的机制，Promise 才能把多个异步步骤自然地串联成一条清晰、线性的异步执行链。

面试题四：Promise 的错误是如何传递的？catch 和 then 的区别是什么？

口述回答模板：

Promise 的错误传递采用的是类似同步异常的“冒泡机制”。
一旦在某个 then 中抛出异常，或者返回了 rejected 状态，这个错误就会沿着 Promise 链一直向后传递，直到被最近的 catch 捕获。

then 方法本身可以接收两个回调参数，分别用于成功和失败处理，但在实际工程中并不推荐这样使用。
catch 本质上等价于 then 的第二个参数，但它语义更清晰，也更适合做统一错误处理。

另外一个很容易被忽略的点是：catch 本身也会返回一个新的 Promise。
如果 catch 中处理了错误并返回了一个正常值，后续的 then 仍然可以继续执行；
如果 catch 中再次抛出异常，错误会继续向后传递。

因此在工程中，通常会在 Promise 链的末尾统一写一个 catch，用来集中处理所有异常情况。

面试题五：Promise 有哪些常用静态方法？分别解决什么问题？

口述回答模板：

Promise 提供了一组静态方法，用于处理多个异步任务的组合场景。

Promise.resolve 用于返回一个成功状态的 Promise，参数可以是普通值，也可以是一个 Promise；如果传入的是一个 Promise 对象，那么 Promise.resolve 不会改变它的状态，而是直接返回一个与该 Promise 状态保持一致的 Promise。

Promise.reject 用于返回一个失败状态的 Promise。

Promise.all 用于并行执行多个 Promise，只有当所有 Promise 都成功时才会成功，只要有一个失败就会整体失败，适合所有结果都必须具备的场景；

Promise.race 返回最先完成的 Promise 结果，不论成功还是失败，常用于超时控制；

Promise.allSettled 会等待所有 Promise 执行完毕，并返回每一个 Promise 的最终状态，不关心成功或失败；

Promise.any 只要有一个 Promise 成功就返回成功结果，只有全部失败才会失败。

这些方法本质上解决的是“多个异步任务如何协同完成”的问题。

面试题六：Promise 和 async / await 的关系是什么？本质区别在哪里？

口述回答模板：

async / await 本质上是 Promise 的语法糖，而不是一种新的异步模型。

async 函数一定会返回一个 Promise，函数内部的 return 会被自动包装成 fulfilled 状态，而抛出的异常会被自动包装成 rejected 状态。

await 的作用是暂停当前 async 函数的执行，等待 Promise 的结果返回，但这种暂停不会阻塞主线程，只会阻塞当前函数后续代码的执行。

从底层机制上看，await 后面的代码依然是通过微任务的方式继续执行的。
在实际开发中，Promise 更适合底层能力封装，而 async / await 更适合编写业务流程代码，两者是配合关系而不是替代关系。

面试题七：Promise 在事件循环中的执行顺序是怎样的？

口述回答模板：

Promise 的 then 和 catch 回调并不会立即执行，而是会被放入微任务队列。

当当前宏任务执行完成、调用栈清空之后，事件循环会优先清空所有微任务，然后才会进入下一轮宏任务。因此，Promise 回调的执行优先级高于 setTimeout 等宏任务。

需要注意的是，then 回调是否进入微任务队列，取决于 Promise 状态是否发生改变。
当 Promise 被 resolve 或 reject 时，对应的回调才会被推入微任务队列，而不是在调用 then 的那一刻立即执行。

这也是理解 Promise 执行顺序和调试异步问题的关键。

会话控制

面试题一：什么是会话控制？为什么需要会话控制？

口述回答模板：

会话控制指的是：在 HTTP 这种无状态协议之上，识别同一个用户的多次请求，并维持其登录态和业务上下文的机制。

HTTP 本身是无状态的，每一次请求在服务器看来都是独立的，如果没有会话控制，服务器无法判断两次请求是否来自同一个用户，也无法知道用户是否已经登录。这在登录校验、购物车、权限控制等场景下是不可接受的。

因此，会话控制的核心目标就是：
在多次请求之间建立“用户身份连续性”，让服务器能够识别用户是谁、是否登录、拥有哪些权限。
Cookie、Session 和 Token，都是围绕这个目标产生的不同实现方案。

面试题二：Cookie 是什么？它在会话控制中起什么作用？

口述回答模板：

Cookie 是浏览器存储的一小段键值对数据，由服务器通过响应头下发，并在后续请求中由浏览器自动携带发送给服务器。

在会话控制中，Cookie 本身并不保存复杂的会话信息，而是承担“客户端标识”的角色。最典型的用法是：服务器生成一个唯一的会话标识，比如 sessionId，通过 Cookie 下发给浏览器，浏览器之后每次请求都会带上这个标识。

Cookie 的优势在于使用简单、浏览器自动管理，但缺点也比较明显：
一是大小和数量受限；
二是安全性依赖配置，如果没有合理设置 HttpOnly、Secure 等属性，容易被窃取或滥用；
三是 Cookie 会自动随请求发送，在跨域和前后端分离场景中不够灵活。

因此，Cookie 更像是会话控制中的“载体”，而不是完整的身份方案。

面试题三：Session 是什么？它和 Cookie 是如何配合工作的？

口述回答模板：

Session 是服务器端的会话存储机制，用来保存用户的登录状态和相关会话数据，比如用户 ID、权限信息等。

典型的 Session 工作流程是：
用户第一次访问时，服务器创建一个 Session，并生成一个唯一的 sessionId；
服务器将这个 sessionId 通过 Cookie 返回给浏览器；
浏览器在后续请求中携带这个 sessionId；
服务器根据 sessionId 查找对应的 Session 数据，从而识别用户身份。

这种模式的核心特点是：会话数据存储在服务器端，客户端只保存一个标识。
它的安全性相对较高，因为敏感信息不暴露在客户端，但也带来了问题，比如服务器需要维护 Session 状态，在分布式或多实例部署时，需要额外的 Session 共享或粘性策略。

面试题四：Token 是什么？为什么前后端分离项目更常用 Token？

口述回答模板：

Token 是一种自包含的身份凭证，通常由服务器在用户登录成功后生成，并返回给客户端。客户端在后续请求中，主动通过请求头携带 Token，而不是依赖浏览器自动发送。

与 Session 不同，Token 模式下服务器通常不保存会话状态，而是通过校验 Token 的合法性来确认用户身份。这种设计使得 Token 天然适合前后端分离和分布式系统。

Token 的优势主要体现在三点：
第一，不依赖 Cookie，跨域和多端使用更灵活；
第二，服务器无状态，天然支持横向扩展；
第三，可以携带用户信息和权限声明，减少服务器查询开销。

当然，Token 也并非没有代价，比如一旦泄露，在过期之前都可能被滥用，因此通常会配合过期时间、刷新机制和 HTTPS 一起使用。

面试题五：Cookie + Session 和 Token 的核心区别是什么？

口述回答模板：

Cookie + Session 属于服务端有状态方案，服务器需要保存会话数据，客户端只保存一个 sessionId；
Token 属于服务端无状态方案，服务器不保存会话，只校验 Token 本身。

前者的优势是安全直观、控制力强，适合传统单体应用；
后者的优势是可扩展性好、跨端友好，适合前后端分离、微服务架构。

本质上，两者解决的是同一个问题：用户身份识别，只是状态存放的位置不同，一个在服务器，一个在客户端。

面试题六：Token 放在哪里？和 Cookie 相比有什么不同？

口述回答模板：

Token 一般不会自动携带，而是由客户端主动放在请求头中，比如 Authorization 字段。这一点和 Cookie 有本质区别。

Cookie 是浏览器自动管理的，适合传统页面应用；
Token 由业务代码控制，更适合移动端、小程序以及前后端分离架构。

但需要注意的是，如果 Token 存储在本地存储中，需要防范 XSS 攻击；如果存储在 Cookie 中，又需要防范 CSRF，因此具体方案需要结合业务场景权衡安全策略。

面试题七：在实际项目中，如何选择 Cookie、Session 还是 Token？

口述回答模板：

选择哪种会话控制方式，本质上取决于系统架构和安全需求。

如果是传统服务端渲染应用，用户规模不大，Cookie + Session 简单可靠；
如果是前后端分离或多端统一登录，Token 更灵活；
如果是分布式或微服务系统，Token 或基于 Token 的统一认证中心更容易扩展。

实际工程中，也经常是混合使用，比如 Cookie 承载 Token，Token 表达身份，结合多种安全策略共同完成会话控制。

面试题八：什么是 JWT？它解决了什么问题？

口述回答模板：

JWT，全称是 JSON Web Token，是一种用于在客户端和服务器之间安全传递身份信息的标准化 Token 格式。
它的核心作用是在无状态的 HTTP 协议之上，实现用户身份认证和授权。

JWT 产生的背景是：在前后端分离和分布式系统中，传统基于 Session 的方案需要服务器保存会话状态，不利于横向扩展。JWT 的思路是把用户身份信息直接放进 Token 中，由客户端保存，每次请求时主动携带，服务器只需要校验 Token 的合法性即可。

因此，JWT 的本质不是“加密技术”，而是一种自包含、可校验、无状态的身份凭证设计方案，非常适合微服务、多端登录和跨域场景。

面试题九：JWT 的结构是怎样的？每一部分有什么作用？

口述回答模板：

JWT 由三部分组成，分别是 Header、Payload 和 Signature，它们通过点号连接成一个字符串。

Header 用来描述 Token 的元信息，比如使用的签名算法类型；
Payload 用来存放实际的业务声明，也就是用户身份信息、权限信息以及一些标准字段，比如过期时间；
Signature 是通过 Header 和 Payload 再结合服务器的密钥计算出来的签名，用于防止 Token 被篡改。

需要特别强调的是：JWT 默认并不是加密的，而只是签名的。
Payload 中的数据是可以被解码查看的，但如果被篡改，签名校验一定会失败，这也是 JWT 安全性的核心所在。

面试题十：JWT 的工作流程是怎样的？

口述回答模板：

JWT 的典型工作流程是这样的：
用户在登录成功后，服务器根据用户信息生成一个 JWT，并返回给客户端；
客户端将这个 JWT 保存起来，比如存在内存或本地存储中；
在后续请求中，客户端通过请求头主动携带 JWT；
服务器接收到请求后，验证 JWT 的签名和有效期，如果校验通过，就认为用户身份是可信的。

整个过程中，服务器不需要保存任何会话状态，这也是 JWT 被称为“无状态认证”的原因。

面试题一：什么是 XSS？它和会话控制有什么关系？

口述回答模板：

XSS，全称是跨站脚本攻击，本质上是攻击者把恶意脚本注入到网页中，让浏览器在用户的上下文里执行这些脚本。

XSS 和会话控制的关系非常直接，因为一旦发生 XSS，攻击者就有机会窃取或滥用用户的会话凭证，比如 Cookie 中的 sessionId，或者本地存储中的 Token。
也就是说，XSS 的真正危害并不只是页面被篡改，而是它可以直接导致用户身份被冒用。

从攻击视角看，XSS 是“绕过服务器认证，直接站在用户角度发请求”；
从防御视角看，任何基于 Cookie、Session 或 Token 的会话控制方案，一旦前端存在 XSS 漏洞，都会面临会话被劫持的风险。

面试题二：XSS 如何影响 Cookie / Session / Token？

口述回答模板：

如果使用的是 Cookie + Session 模型，并且 Cookie 没有设置 HttpOnly，那么 XSS 脚本可以直接读取 Cookie 中的 sessionId，并发送给攻击者，从而实现会话劫持。

如果使用的是 Token，并且 Token 存储在 localStorage 或 sessionStorage 中，那么 XSS 脚本同样可以直接读取 Token，这在前后端分离项目中尤为常见。

需要强调的是：XSS 并不会绕过认证逻辑，而是窃取合法身份凭证后再正常使用它。
因此，从安全角度看，XSS 是所有会话控制方案中最优先需要防范的一类攻击。

面试题三：什么是 CSRF？它和会话控制的关系是什么？

口述回答模板：

CSRF，全称是跨站请求伪造，它的核心特点是：攻击者无法读取用户信息，但可以借助用户已登录的身份发起请求。

CSRF 的前提是浏览器会自动携带身份凭证，比如 Cookie。
如果用户在登录状态下访问了恶意网站，该网站就可以构造请求，利用浏览器自动携带 Cookie 的机制，向目标服务器发送请求，从而以用户身份执行操作。

因此，CSRF 的攻击目标并不是窃取会话，而是滥用已有会话。
它与会话控制的关系在于：只要身份凭证会被浏览器自动发送，就天然存在 CSRF 风险。

面试题四：为什么 Cookie + Session 容易受到 CSRF 攻击，而 Token 相对不容易？

口述回答模板：

Cookie + Session 模式下，浏览器会在符合域名条件时自动携带 Cookie，这正是 CSRF 能成立的根本原因。
攻击者不需要知道 sessionId，只需要诱导用户发起请求，浏览器就会自动附带身份信息。

而 Token 通常通过请求头由前端代码主动携带，而不是浏览器自动附带。
恶意网站无法在不触发同源限制的情况下构造携带 Token 的请求，因此 Token 天然规避了大多数 CSRF 攻击场景。

需要注意的是，这并不意味着 Token 完全免疫 CSRF，而是Token 的携带方式让 CSRF 的攻击成本显著提高。

面试题五：如何防御 XSS？从会话安全角度如何设计？

口述回答模板：

防御 XSS 的核心原则是：永远不要信任用户输入。
在前端和后端都需要对用户输入进行转义或过滤，避免脚本被当作可执行代码插入页面。

从会话控制角度，最重要的防御措施包括：
对 Cookie 设置 HttpOnly，防止脚本读取；
对 Token 避免暴露在容易被脚本读取的位置；
启用 Content Security Policy，限制脚本加载来源。

可以总结为一句话：XSS 一旦成立，会话控制基本失效，因此必须在源头阻断 XSS。

面试题六：如何防御 CSRF？常见方案有哪些？

口述回答模板：

防御 CSRF 的核心思路是：让请求具备“攻击者无法伪造”的特征。

最常见的方案是 CSRF Token，即服务器为每个用户生成一个随机值，要求客户端在请求中主动携带，并在服务器端校验。
由于攻击者无法读取页面内容，因此也无法构造合法的 CSRF Token。

另外，合理设置 Cookie 的 SameSite 属性，也可以在浏览器层面阻止跨站请求自动携带 Cookie。
在前后端分离项目中，使用 Token 并通过请求头携带，本身就是一种有效的 CSRF 防御手段。

前端模块化

面试题：什么是前端打包工具？

前端打包工具（Bundler / Build Tool）的核心目标是：

将分散的源代码与资源，转换、整合并输出为浏览器或运行环境可理解的高性能生产代码。

打包工具到底“打包”了什么

打包工具处理的不是只有 JS，而是整个前端工程的所有输入：

  • JavaScript / TypeScript
  • CSS / SCSS / Less
  • 图片 / 字体 / SVG
  • JSON / WASM
  • 环境变量 / 配置文件

最终目标只有一个：

生成一组浏览器能够高效加载、稳定运行、长期缓存的静态资源。

面试题：前端打包工具的作用是什么？

前端打包工具本质上解决三类问题。
第一是工程化问题，现代前端采用模块化开发，但浏览器无法理解工程结构，需要打包工具分析模块依赖并整合代码。
第二是兼容性问题，工程中使用的 TypeScript、SCSS 以及新语法浏览器无法直接识别，需要在打包阶段进行转换和降级。
第三是性能问题，工程化拆分会带来大量文件和请求，打包工具通过合并、拆分和按需加载等方式，在构建阶段优化最终加载性能。

面试题：前端打包工具的流程是什么？

从入口文件开始，先分析模块之间的依赖关系，构建完整的依赖图；
然后对每个模块进行代码转换，把工程化语言转成浏览器可执行代码；
接着根据依赖关系决定代码如何合并或拆分；
在此基础上进行构建阶段的优化，比如压缩和删除无用代码；
最后生成浏览器可以直接加载的最终产物。

面试题：常见的前端打包工具有哪些？

常见的前端打包工具包括 Webpack、Vite、Rollup、esbuild 和 Parcel。Webpack 功能最全面，适合复杂工程；Vite 基于原生 ESM，开发体验最好；Rollup 更适合打包类库，输出干净；esbuild 以极致性能著称，常作为底层构建工具；Parcel 强调零配置，适合小型项目和快速原型。

面试题：什么是Webpack？

Webpack 是一个以模块为核心的前端构建工具，其本质是：

从一个或多个入口文件出发，构建整个应用的依赖图（Dependency Graph），并将其打包为若干个可部署的静态资源。

Webpack 的核心思想

  • 一切皆模块 JS/CSS/图片/字体/JSON
  • 模块之间通过 import / require 建立依赖
  • Webpack 递归分析依赖，最终形成资源图

面试题：什么是 loader？为什么需要 loader？

口述回答模板：

在 Webpack 中，loader 本质上是一个文件转换器。
因为 Webpack 本身只认识 JavaScript 和 JSON，而在真实项目中，我们会使用 CSS、图片、字体、TypeScript、JSX、Vue 文件等各种非 JS 资源，所以需要 loader 把这些资源转换成 Webpack 能理解和处理的模块。

从工程角度看，loader 解决的问题是：
如何把不同类型的资源，统一纳入模块依赖图中进行打包处理。
每一个 loader 只做一件事，比如编译、转换、提取或注入，多个 loader 还可以串联形成流水线。

需要注意的是，loader 工作在构建阶段，不参与运行时，它的目标是生成最终可执行的模块代码。

面试题：loader 的执行机制是怎样的？

口述回答模板：

loader 的执行是从右到左、从下到上的，也可以理解为“先处理源代码，再逐步包装结果”。
每一个 loader 接收上一个 loader 的输出作为输入，最终生成 Webpack 可识别的模块。

这种设计让 loader 具备很强的组合能力，比如先做语法转换，再做样式处理，再做资源注入，每一步职责都很清晰。

面试题：常见的 loader 有哪些？分别解决什么问题？

口述回答模板：

在实际项目中，常见 loader 可以按资源类型来理解。

  • 处理 JavaScript 和语法转换的，最典型的是 babel-loader，用来把 ES6+ 代码转换成兼容性更好的 JavaScript；
  • 处理 TypeScript 的是 ts-loader 或 babel-loader 配合 TypeScript 插件，用于类型编译和语法转换；
  • 处理 CSS 的是 css-loader，它负责解析 CSS 中的依赖关系，比如 @import 和 url；style-loader 用于把 CSS 通过 style 标签注入到页面中，而在生产环境中，通常会用 MiniCssExtractPlugin 配合 loader 把 CSS 抽离成独立文件；
  • 预处理器相关的有 sass-loader、less-loader，用于把 Sass、Less 编译成标准 CSS；
  • 处理图片和字体等静态资源的有 file-loader、url-loader，在 Webpack 5 中更多由内置的 asset 模块替代；
  • 在 Vue 项目中，vue-loader 用于解析 .vue 单文件组件，把 template、script、style 拆分并分别交给对应 loader 处理。

这些 loader 各司其职，共同完成从源码到最终资源的转换过程。

面试题：loader 和 plugin 有什么区别？

口述回答模板：

loader 和 plugin 的核心区别在于职责不同。

loader 关注的是“某一类文件如何被转换”，它作用于模块本身，通常是单输入单输出；
plugin 关注的是“整个构建过程如何被增强或控制”，它可以介入打包的任意生命周期，比如资源优化、文件生成、环境注入等。

可以简单理解为：
loader 是流水线上的加工工人，
plugin 是控制整个流水线流程的调度器。

面试题：plugin 的工作原理是什么？

口述回答模板：

plugin 的核心原理是：基于事件机制的生命周期钩子系统。

Webpack 在内部会在关键阶段触发不同的钩子，plugin 在初始化时把自己的逻辑注册到这些钩子上。
当构建流程执行到对应阶段时，Webpack 会调用 plugin 提供的回调，从而让 plugin 有机会读取、修改或新增构建过程中的数据。

从实现角度看，一个 plugin 本质上是一个带有 apply 方法的对象，Webpack 会在启动阶段调用这个 apply 方法，并传入 compiler 对象，plugin 再通过 compiler 或 compilation 上的钩子介入构建。

面试题：常见的 plugin 有哪些？分别解决什么问题？

口述回答模板：

在实际项目中，常见的 plugin 往往围绕构建优化和工程能力展开。

比如 HtmlWebpackPlugin 用于根据模板自动生成 HTML 文件，并注入打包后的资源；
DefinePlugin 用于在构建阶段注入全局常量，常用于区分开发和生产环境；
MiniCssExtractPlugin 用于将 CSS 从 JS 中抽离成独立文件；
CleanWebpackPlugin 用于在每次构建前清理输出目录；
CopyWebpackPlugin 用于拷贝静态资源；
BundleAnalyzerPlugin 用于分析打包结果，帮助优化体积。

这些 plugin 的共同特点是：它们并不关心单个文件内容，而是关注整个构建产物和流程。

面试题：什么时候应该写 plugin，而不是 loader？

口述回答模板：

当需求只是“如何处理某一类文件”时，应该使用 loader；
当需求涉及“构建流程控制、资源生成、全局注入、打包结果处理”时，就必须使用 plugin。

比如：
编译一种新语法用 loader；
根据打包结果生成文件、统计信息或优化策略，就需要 plugin。

面试题：你对 Babel 的理解是什么？它解决了什么问题？

口述回答模板：

Babel 是一个 JavaScript 编译器，核心作用是把高版本或非标准的 JavaScript 语法，转换成兼容性更好的 JavaScript，从而让代码能够在更多浏览器和运行环境中正常运行。

在前端工程中，我们希望使用最新的语言特性来提升开发效率和代码可读性，但实际运行环境可能并不支持这些新语法。Babel 的价值就在于把“开发阶段的现代语法”和“运行阶段的兼容代码”解耦，让开发体验和兼容性可以同时兼顾。

面试题：Babel 的工作原理是怎样的？

口述回答模板：

Babel 的编译过程可以概括为三个阶段：解析、转换和生成。

首先是解析阶段，Babel 会把源代码解析成抽象语法树，也就是 AST；
接着是转换阶段，通过一系列插件对 AST 进行遍历和修改，把新语法结构转换成旧语法对应的 AST 结构；
最后是生成阶段，根据修改后的 AST 重新生成目标 JavaScript 代码。

理解这三步非常重要，因为 Babel 的所有能力，本质上都是在“如何分析和修改 AST”。

面试题：什么是 Webpack 的热加载？它解决了什么问题？

口述回答模板：

Webpack 的热加载，通常指的是 Hot Module Replacement，也就是 HMR。
它的核心作用是在开发环境中，当模块代码发生变化时，只替换发生变化的模块，而不是刷新整个页面。

在没有热加载的情况下，代码一改动，浏览器就会整体刷新页面，应用状态会全部丢失，比如表单输入、滚动位置、组件内部状态等，这会严重影响开发效率。
而 HMR 的目标是：尽量保留页面当前状态，只更新必要的代码，让开发反馈更快、更接近真实交互。

面试题：Webpack 热加载的工作原理是什么？

口述回答模板：

Webpack 热加载的本质是：构建阶段 + 浏览器运行时之间的一套模块替换机制。

在开发模式下，Webpack 会启动一个开发服务器，并在构建时为每个模块维护依赖关系。当某个模块的源码发生变化时，Webpack 会重新编译这个模块以及它受影响的依赖，而不是重新打包整个应用。

编译完成后，开发服务器通过 WebSocket 等方式，把“模块更新信息”推送给浏览器。
浏览器端的 HMR runtime 会根据这些信息，找到对应的旧模块，并尝试用新模块进行替换。

如果模块本身或它的父模块显式接受这次更新，那么替换就会成功，页面不会刷新；
如果没有任何模块能接受更新，Webpack 才会退化为整页刷新。

面试题：Webpack 是如何优化前端性能的？

口述回答模板：

Webpack 对前端性能的优化，本质上发生在构建阶段，目标是生成体积更小、加载更快、执行更高效的资源，从而提升页面加载和运行性能。

首先是代码体积层面的优化。
Webpack 通过生产模式下的压缩与 Tree Shaking，移除未使用的代码，减少最终 bundle 的体积。
Tree Shaking 基于 ES Module 的静态分析能力，只保留真正被引用的导出，这对第三方库和工具函数的体积优化非常关键。

其次是代码拆分和按需加载。
Webpack 支持 Code Splitting，可以把应用拆成多个 chunk，把首屏不需要的代码延后加载。
典型场景包括路由懒加载、组件异步加载等，这能显著降低首屏 JS 体积，提升首屏渲染速度。

第三是缓存友好的构建策略。
Webpack 可以通过内容哈希来为资源生成稳定的文件名，当代码不变时文件名不变，浏览器就能命中缓存。
同时通过拆分 runtime、vendor 和业务代码，可以避免一次小改动导致整个 bundle 失效，提高长期缓存命中率。

第四是资源处理与加载优化。
Webpack 可以压缩图片、抽离 CSS、合并和优化静态资源，并结合 preload、prefetch 等机制，合理控制资源加载的优先级，让关键资源更早加载，非关键资源延后加载。

第五是构建效率和开发体验的间接优化。
在开发阶段，通过 HMR 热更新、缓存 loader 结果、多进程打包等方式，减少构建等待时间，提高开发效率，从而加快迭代节奏，这虽然不直接影响线上性能，但对整体工程效率非常重要。

最后，通过 构建结果分析工具，比如分析 bundle 体积和依赖结构，持续发现性能瓶颈，形成性能优化的闭环。

面试题：在 Webpack 中，module、chunk、bundle 分别是什么？它们之间是什么关系？

口述回答模板：

在 Webpack 中，module、chunk 和 bundle 是三个不同层级的概念，分别对应源码层、构建层和产物层。

首先是 module。
module 是 Webpack 能处理的最小单元，本质上就是“模块”。
在项目中，一个 JS 文件、一个 CSS 文件、一张图片，经过 loader 处理后，都会被当作一个 module。
module 关注的是源码层面的依赖关系，也就是谁 import 了谁，谁依赖了谁。

接下来是 chunk。
chunk 是 Webpack 在构建过程中，根据模块依赖关系和代码拆分规则，把多个 module 组合在一起形成的一个中间产物。
chunk 并不是最终输出的文件，而是 Webpack 用来做打包、拆分、优化的逻辑单位。
比如入口 chunk、异步 chunk、vendor chunk，本质上都是由一组 module 组成的。

最后是 bundle。
bundle 是 Webpack 最终输出到磁盘上的文件，也就是浏览器真正加载的 JS 或 CSS 文件。
通常情况下，一个 chunk 会对应生成一个或多个 bundle 文件，但 bundle 一定是构建完成后的结果。

面试题：chunk 是什么时候产生的？由什么决定？

口述回答模板：

chunk 是在 Webpack 构建阶段产生的。
它的生成规则由入口配置、动态 import、代码拆分策略以及优化配置共同决定。

比如：
每一个 entry 默认会生成一个入口 chunk；
通过动态 import 引入的模块，会生成异步 chunk；
通过 splitChunks 配置，Webpack 还可以把公共模块抽成独立的 chunk。

因此，chunk 是 Webpack 做性能优化和代码拆分的核心载体。

面试题：bundle 和 chunk 是一一对应的吗？

口述回答模板：

在大多数情况下，一个 chunk 会对应一个 bundle 文件，但并不是绝对的一一对应。

比如在开启 CSS 抽离时，一个 chunk 可能会同时生成 JS bundle 和 CSS bundle；
在某些优化场景下，也可能出现多个 chunk 合并成一个 bundle 的情况。

所以更准确地说是：
chunk 决定了“逻辑拆分”，bundle 决定了“物理输出”。

面试题：单页应用是一个 bundle 还是多个 chunk？

口述回答模板：

单页应用在概念上是“一个应用、一个入口页面”，但在构建结果上，并不等于只有一个 bundle。
更准确地说，单页应用通常是：一个入口 + 多个 chunk + 多个 bundle。

从 Webpack 的角度看，单页应用通常只有一个入口 entry，这个入口会生成一个入口 chunk。
如果不做任何优化，所有代码都会被打进这个入口 chunk，最终输出一个 bundle，这种情况确实是“一个 SPA 一个 bundle”。

但在实际工程中，为了性能优化，单页应用几乎一定会做代码拆分。
比如路由懒加载、组件异步加载、第三方库拆分，这些都会让 Webpack 在构建阶段生成多个 chunk。

这些 chunk 在输出阶段会对应生成多个 bundle 文件，浏览器会根据需要按需加载。
因此，大多数现代单页应用在运行时，页面初始只加载必要的 bundle，后续再动态加载其他 bundle。

面试题：什么是 Vite？它解决了什么问题？

口述回答模板：

Vite 是一个现代前端构建工具，核心目标是大幅提升开发阶段的启动速度和更新速度。
它的名字来源于法语，意思是“快”，这也正是它设计的出发点。

在传统构建工具中，比如 Webpack，开发服务器启动前需要先把整个项目打包成 bundle，项目越大，启动和热更新就越慢。
Vite 的创新点在于：在开发阶段不做整体打包，而是基于浏览器原生 ES Module 的能力，按需加载源码。

因此，Vite 把“开发体验”和“生产构建”这两个阶段彻底拆开，分别采用最适合的技术方案。

面试题：Vite 的核心工作原理是什么？

口述回答模板：

在开发环境下，Vite 并不会先打包整个应用。
它启动的是一个基于原生 ESM 的开发服务器，当浏览器请求某个模块时，Vite 才即时编译并返回这个模块。

这种方式让冷启动速度几乎只和“启动服务器”有关，而和项目体积关系不大。
当代码发生变化时，Vite 只需要重新处理被修改的模块，并通过模块级别的热更新推送给浏览器，更新速度非常快。

而在生产环境中，Vite 并不会自己造轮子，而是使用成熟的打包工具来完成构建，确保输出的代码体积和性能。

面试题：Vite 和 Webpack 的核心区别是什么？

口述回答模板：

Vite 和 Webpack 的最大区别在于开发阶段的构建模型不同。

Webpack 在开发和生产阶段，都是以“打包”为核心思路；
Vite 在开发阶段是“按需编译 + 原生 ESM”，在生产阶段才进行完整打包。

这也是为什么在大型项目中，Vite 的启动速度和热更新体验明显优于传统打包工具，而在生产构建质量上又能保持可靠性。

面试题：Vite 为什么热更新这么快？

口述回答模板：

Vite 的热更新是基于 ES Module 的模块级更新，而不是 bundle 级别的更新。

当一个模块发生变化时，Vite 只会重新编译这个模块，并通知浏览器精确更新对应的依赖关系，而不是重新生成整个依赖图。
这种“最小更新单元”的设计，使得热更新的耗时几乎与项目规模无关。

面试题：vite完整流程

【完整请求链路】

假设入口是 index.html，其中有：

浏览器的行为与 Vite 的行为按时间顺序是：

  1. 浏览器请求 GET /index.html
    • Vite 返回 HTML
    • 同时会在 HTML 中注入 HMR 客户端脚本（用于热更新通信）
  2. 浏览器看到 <script type="module" src="/src/main.ts">
    • 请求 GET /src/main.ts
  3. Vite 收到 /src/main.ts 请求
    • 如果是 TS：用 esbuild 做快速转译成 JS（不做类型检查）
    • 如果里面有 import：Vite 会把 import 路径改写成浏览器可请求的 URL 形式（重要）
    • 返回转译后的 JS 给浏览器
  4. 浏览器执行并解析 import，例如：
    • import App from './App.vue' → 浏览器会请求 GET /src/App.vue
    • import { createApp } from 'vue' → 浏览器会请求一个被改写后的依赖地址（通常带 /@modules/ 或 /node_modules/.vite/deps/... 的形式，取决于版本与实现）
  5. Vite 针对不同资源类型做不同处理并返回
    • .vue：走 Vue 插件，把 SFC 拆成多个虚拟模块（template/script/style），再分别返回
    • .css：转换成 JS 模块（用于插入样式或 CSS Modules 处理），并建立 HMR 依赖
    • 图片字体等：返回 URL 或内联数据（取决于大小与配置）
    • 第三方依赖：优先走“依赖预构建”后的产物（下面详讲）
  6. 浏览器继续递归请求所有 import 依赖
    • Vite 继续按需编译与返回
    • 没有“全量打包”这一步

这就是 Vite 开发阶段启动快的根源：启动时不需要扫描整个项目并生成 bundle，只要起服务即可。

面试题：Vite 是如何处理各种前端资源的？

口述回答模板：

Vite 对资源的处理核心思路是：开发阶段尽量不打包、按需处理；生产阶段再统一打包和优化。
不同类型的资源，在这两个阶段的处理方式是不同的。

在开发阶段，Vite 基于浏览器原生 ES Module。
JavaScript 和 TypeScript 源码会在浏览器请求时被即时编译并返回，而不是提前打包成 bundle。
对于 TypeScript，Vite 只做语法层面的转译，不做类型检查，从而保证启动和更新速度。

对于 CSS 资源，Vite 会把 CSS 当作模块处理。
浏览器请求到 CSS 时，Vite 会直接返回编译后的 CSS，并通过 HMR 实现样式的局部热更新，更新时不会刷新页面，也不会丢失应用状态。
预处理器如 Sass、Less 会在请求时被即时编译成 CSS。

对于 图片、字体等静态资源，Vite 会把它们当作 URL 模块来处理。
在开发阶段，资源通常以原始路径或带 hash 的 URL 形式被引用，不做复杂的合并和压缩，保证请求链路简单、调试友好。

对于 JSON 等非 JS 资源，Vite 会自动将其转换成 ES Module，让它们可以被直接 import 使用。

当进入生产构建阶段时，Vite 会把整个项目交给底层的打包器进行统一处理。
这一步会完成代码压缩、Tree Shaking、资源合并、代码拆分、hash 命名、CSS 抽离等优化操作，生成适合线上部署的 bundle 产物。

因此可以总结为：
开发阶段，Vite 更像是一个“按需编译的资源服务器”；
生产阶段，Vite 才真正完成“构建和打包”的工作。
