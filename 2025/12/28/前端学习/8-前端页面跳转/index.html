前端页面跳转

1. SPA（单页应用）

SPA 的核心思想不是“页面只有一页”，而是：

整个应用生命周期中，浏览器只加载一次 HTML 文档，之后的“页面切换”不再由浏览器完成，而是由前端 JavaScript 接管。

应用周期：从浏览器首次加载并执行应用入口代码开始，到当前文档被销毁为止，前端应用在同一 HTML 文档内持续运行的整个时间段。

只要发生文档级导航，应用周期立即结束：

  • 刷新页面（F5）
  • 地址栏输入新 URL 回车
  • 跳转到另一个站点
  • 当前 tab 关闭

结束意味着：

  • HTML 被销毁
  • JS 上下文清空
  • 所有 Router / Store / 组件全部消失

一、先从“传统网页”说起

在传统 Web（也叫 MPA，多页应用）中：

  1. 每次点击链接或提交表单
  2. 浏览器都会发起一次新的 HTTP 请求
  3. 服务器返回一个新的 HTML 文档
  4. 浏览器销毁旧页面，重新解析 HTML、CSS、JS
  5. 页面重新渲染

也就是说：

  • 页面跳转 = 浏览器行为
  • 页面内容 = 服务器渲染结果
  • 每一次跳转，都是一次“文档级刷新”

二、SPA 的根本改变是什么

SPA 把这套逻辑彻底反过来了。

在 SPA 中：

  1. 浏览器第一次访问站点
  2. 服务器返回 一个基础 HTML（通常是 index.html）
  3. 浏览器加载并执行大量 JS
  4. 之后的所有“页面切换”：
    • 不再请求新的 HTML
    • 不刷新页面
    • 只通过 JS 改变视图内容

也就是说：

  • 浏览器只负责 加载一次壳
  • 前端 JS 负责 整个应用运行期的页面管理

三、为什么叫“单页”，但却能有很多页面

SPA 中常见的现象是：

看起来像多个页面，但实际上：

  • 这些 URL 并不对应多个 HTML 文件
  • 它们只是：
    • 同一个 HTML
    • 在不同 URL 状态下
    • 渲染不同的组件树

“单页”指的是：

HTML 文档层面是单一的

而不是“视觉上只能有一个界面”

四、SPA 中“页面切换”到底发生了什么

以 Vue SPA 为例，一次路由切换本质上是：

  1. URL 发生变化（hash 或 history）
  2. Vue Router 监听到变化
  3. Router 解析当前路径
  4. 匹配到某个路由规则
  5. 决定哪些组件要挂载 / 卸载
  6. Vue 更新虚拟 DOM
  7. 局部 DOM 发生变化

整个过程没有浏览器级刷新。

五、SPA 必须依赖哪些关键能力

SPA 并不是凭空出现的，它依赖三类浏览器能力：

  1. JavaScript 长时间运行

JS 不再是“点一下执行一下”，而是：

  • 应用启动即常驻
  • 控制状态、路由、渲染、交互
  1. 浏览器 URL 可被“劫持解释”

浏览器允许：

  • hash 不触发刷新
  • History API 修改地址栏不刷新

这使得：

  • URL 可以作为“状态表达”
  • 但页面不会重载
  1. DOM 可被 JS 动态重构

SPA 的页面不是 HTML 写死的，而是：

  • 数据 → 状态 → 组件 → DOM
  • DOM 是 JS 计算出来的结果

浏览器做的第一件事都是解析 URL，拆成固定的结构：

  • scheme（协议）
  • host（域名）
  • port（端口）
  • path
  • query
  • fragment（也就是 hash）

这是标准化行为，不依赖前端框架。

接下来浏览器要做一个非常关键的判断：

“这次 URL 变化，是否需要重新请求文档？”

这个判断，几乎完全由 path / query / fragment 是否变化 决定。

为什么 hash 永远不会触发请求:

浏览器在网络层有一个硬性规则：

fragment 不参与 HTTP 请求

也就是说，URL 在真正发往服务器之前，浏览器会直接把 # 后面的内容丢弃。

例如你访问：

浏览器在网络层看到的是：

服务器永远不知道你写了 #/user?tab=1。

这不是 Vue Router 的设计，而是 HTTP + URL 规范。

hash 变化为什么不刷新页面

当浏览器发现：

  • scheme 没变
  • host 没变
  • path 没变
  • query 没变
  • 只变了 fragment

浏览器会把这次跳转判定为：

同一文档内的导航（same-document navigation）

同文档导航意味着：

  • 不重新请求 HTML
  • 不重新解析 DOM
  • 不重新执行 JS
  • 仅更新地址栏
  • 更新历史记录

这是 hash 能被 SPA 利用的核心前提。

浏览器对 hash 的“原生职责”

在 SPA 出现之前，hash 就已经存在了，它本来就有三层语义。

第一层语义是文档锚点。
如果 hash 对应一个 DOM 元素的 id 或 name，浏览器会自动滚动到该位置。这是 HTML 规范级行为，不需要 JS 参与。

第二层语义是状态标记。
hash 的变化不会触发请求，但会反映在地址栏中，因此它天然适合作为“客户端状态”的外显形式。SPA 正是利用这一点，把 #/user/123 当作“当前视图状态”。

第三层语义是历史记录节点。
即使只是改 hash，浏览器也会新增一条 history entry，因此 back / forward 都是可用的。

当 fragment 变化时，浏览器会在内部完成三件事：

  1. 更新地址栏
  2. 写入 history
  3. 在当前执行栈清空后，派发 hashchange 事件

Vue Router 的 hash 模式，本质上就是：

  • 修改 location.hash
  • 监听 hashchange
  • 根据新的 hash 做路由匹配
  • 更新组件树

Router 并没有“控制浏览器”，只是利用浏览器已经定义好的行为。

Vue Router 在 history 模式下到底做了什么

当页面首次加载时，流程是这样的：

  1. 浏览器请求 /user/123
  2. 服务器返回 index.html
  3. 浏览器解析 HTML，加载 JS
  4. Vue 应用启动
  5. Vue Router 读取当前浏览器地址：
  1. Router 把 pathname 当作“当前路由”
  2. 做路由匹配
  3. <router-view> 渲染对应组件

注意一个关键点：

history 模式下，Router 并不会“跳转”

它只是“读取当前 URL，然后解释它”

浏览器对 pushState 的规则是：

  • 修改地址栏
  • 写入 history
  • 不发请求
  • 不刷新页面
  • 只在 back / forward 时触发 popstate

Router 正是通过监听 popstate，来处理浏览器前进 / 后退。

为什么 hash 会写 history，但不触发 popstate

hash 的行为本质

当你修改 hash 时，浏览器做的是：

  1. 在当前 history entry 后面新增一个 entry
  2. 当前指针移动到新 entry
  3. 触发 hashchange

这一步不是在“激活已有 entry”，而是在创建新 entry。

popstate 的唯一触发条件

popstate 只在“激活已有 history entry”时触发

也就是：

  • 从 entry C → 回到 entry B
  • 或从 entry B → 前进到 entry C

而不是“新建 entry”。

为什么 hash 后退也不触发 popstate

即便是：

后退到 /#b：

  • 浏览器认为：这是 hash 导航体系
  • 事件仍然归类给 hashchange
  • popstate 仍然不触发

这是规范中的刻意分工，不是 bug。

pushState 为什么“什么都不触发”

  • 不触发 popstate
  • 不触发 hashchange
  • 不刷新页面
  • 只改 URL + history

这就是为什么：

Router 在调用 pushState 前，必须自己先更新内部状态

浏览器后退时，事件是怎么分流的

情况 A：history 模式（path/query 变化）

后退：

  • 激活已有 history entry
  • 触发 popstate
  • Router 监听它，同步状态

情况 B：hash 模式

后退：

  • hash 变化
  • 触发 hashchange
  • 不触发 popstate

Router hash 模式监听的是 hashchange，不是 popstate。

直接记下面这四条：

  1. hash 变化：写 history，永远走 hashchange
  2. pushState：写 history，但不触发任何事件
  3. popstate：只在激活已有 history entry 时触发
  4. 事件不是“history 改了就触发”，而是“谁改的，走谁的事件”

映射回 Vue Router

Router 模式 路由状态放哪 监听事件 触发场景
hash fragment hashchange hash 改变 / hash 后退
history history entry popstate 浏览器前进 / 后退

而在 router.push() / router.replace() 时：

  • 两种模式都不靠浏览器事件
  • 都是 Router 主动更新内部响应式状态
  • 事件只用于“用户绕过 Router 的行为”
