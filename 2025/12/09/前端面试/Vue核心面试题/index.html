Vue核心面试题

Vue 和 React的核心区别是什么？

Vue 和 React 的核心区别主要在定位和设计理念上。

Vue 更偏向一个完整框架，官方提供了比较完善的解决方案，比如路由、状态管理等，上手成本低、开发规范统一，比较适合快速开发和团队协作。

React 更偏向一个视图库，主要负责 UI 渲染，其它能力需要开发者自行选择和组合，灵活性更高，但对工程能力要求也更高。

从使用感受上看，Vue 对初学者更友好，React 更适合大型、长期维护的复杂项目。

面试题：介绍一下 Vue 的生命周期

口述回答模板：

Vue 的生命周期指的是：一个组件从创建、挂载、更新到销毁的完整过程。
Vue 在这个过程中，在关键阶段暴露了一系列生命周期钩子，让开发者可以在合适的时机执行逻辑，比如初始化数据、发起请求、操作 DOM、清理副作用等。

从整体上看，Vue 的生命周期可以分为四个阶段：创建、挂载、更新和销毁。

在创建阶段，组件实例刚被创建，但 DOM 还没有生成。
这个阶段的特点是：数据已经初始化，但还不能访问真实 DOM。
通常在这里做一些与 DOM 无关的初始化工作，比如初始化数据结构、读取本地缓存、设置定时器配置等。

接下来是挂载阶段。
在这个阶段，Vue 会把模板编译成真实 DOM，并挂载到页面中。
当组件真正挂载完成后，就可以安全地访问和操作 DOM 了。
因此，涉及 DOM 读取、第三方库初始化、依赖页面结构的逻辑，通常都放在挂载完成之后执行。

第三个阶段是更新阶段。
当组件的响应式数据发生变化时，Vue 会触发重新渲染流程。
在更新前和更新后，Vue 都提供了对应的生命周期钩子，让我们可以在数据变化前后做一些处理，比如对比新旧数据、在更新后获取最新的 DOM 状态等。

需要注意的是，这个阶段可能会被多次触发，因此不适合在这里做高成本或一次性的逻辑。

最后是销毁阶段。
当组件从页面中被移除时，会进入销毁流程。
在这个阶段，Vue 会解绑事件、停止响应式追踪，并释放资源。
开发者需要在这里手动清理一些副作用，比如定时器、全局事件监听、订阅关系等，防止内存泄漏。

响应式原理

二、面试题一：什么是 Vue 的响应式原理？

口述回答模板：

Vue 的响应式原理指的是：当数据层的状态发生变化时，框架能够自动感知这个变化，并驱动视图层进行更新，而不需要开发者手动操作 DOM。

从整体模型来看，它体现的是一种数据驱动视图的思想，也就是 Model 变化 → View 自动更新。
Vue 在中间做的事情，是建立数据和视图之间的依赖关系，并在数据变化时，精准通知相关视图进行更新。

三、面试题二：Vue 响应式的核心思想是什么？

口述回答模板：

Vue 响应式的核心思想可以概括为两点：数据劫持 + 依赖收集与派发更新。

数据劫持负责“感知变化”，也就是在数据被读取或修改时能够被拦截；
依赖收集和派发更新负责“通知变化”，也就是知道哪些地方用到了这个数据，并在数据变化时通知它们重新执行。

这套机制保证了 Vue 能做到精确更新，而不是全量刷新视图。

四、面试题三：Vue 2 的响应式原理是如何实现的？

口述回答模板：

在 Vue 2 中，响应式是通过 Object.defineProperty 来实现的。

Vue 会在初始化阶段，遍历 data 中的每一个属性，为它们定义 getter 和 setter。
当属性被读取时，会触发 getter，在 getter 中完成依赖收集，把当前使用这个数据的 Watcher 记录下来；
当属性被修改时，会触发 setter，在 setter 中进行派发更新，通知所有依赖这个数据的 Watcher 重新执行，从而触发视图更新。

在渲染过程中，模板中用到的数据都会触发 getter，因此 Vue 能自动建立“数据 → Watcher → 视图”的依赖关系。

五、面试题四：Vue 2 响应式有什么缺陷？为什么？

口述回答模板：

Vue 2 响应式的缺陷，根源在于 Object.defineProperty 的能力限制。

首先，它只能拦截已经存在的属性，如果后续给对象新增或删除属性，是无法触发 getter 和 setter 的；
其次，对于数组，它无法拦截通过索引直接赋值的行为，比如 arr[0] = xxx，因为这相当于新增属性。

因此在 Vue 2 中，需要通过特定 API，比如 $set，或者通过数组变异方法，来保证响应式更新。

六、面试题五：Vue 3 的响应式原理和 Vue 2 有什么不同？

口述回答模板：

Vue 3 在思想上和 Vue 2 是一致的，仍然是数据劫持 + 依赖收集 + 派发更新，但底层实现发生了变化。

Vue 3 使用的是 Proxy 和 Reflect，通过代理整个对象来拦截操作。
由于 Proxy 是基于对象本身进行代理的，它可以天然拦截属性的读取、修改、新增和删除操作，因此彻底解决了 Vue 2 中新增属性和数组索引不响应的问题。

七、面试题六：Vue 3 的 Proxy 是深度代理吗？

口述回答模板：

Vue 3 采用的是懒代理机制，而不是一次性递归代理所有嵌套对象。

也就是说，只有当访问某个嵌套对象时，Vue 才会在那一刻对它进行 Proxy 包装。
这种设计可以显著减少初始化阶段的性能开销，同时保证访问到的对象一定是响应式的。

Vue模板解析的过程

1. 总览：

核心思想：**View = f(state)**，Vue 用响应式系统知道“什么时候重新算 f”，用 VDOM+Diff 知道“怎么算得更省”。

2. Template 到 Render：Vue 编译做了什么

2.1 编译阶段的 3 步（Template → AST → Render）

  • Parse：把模板字符串解析成 AST（抽象语法树）
  • Optimize（Vue2 典型）：标记静态节点/静态根（不依赖数据的部分）
  • Generate：生成 render() 字符串代码，再 new Function 得到真正的 render 函数

2.2 render() 大概长什么样（理解即可）

render 的本质就是一堆 _c() / h() 调用，返回 VNode：

  • Vue2：_c('div', data, children) 生成 VNode
  • Vue3：h('div', props, children)（并且编译器会加很多 PatchFlag 做优化）

你只要记住：模板不是直接渲染 DOM，而是先被编译成“创建 VNode 的函数”。

3. 虚拟 DOM（VNode）到底是什么

3.1 VNode 的本质

VNode 是一个 JS 对象，描述真实 DOM 节点所需的信息，例如：

  • tag：标签名（div、span…）
  • data/props：属性、事件、指令相关信息
  • children：子 VNode 列表
  • text：文本节点内容
  • key：用于列表 Diff 的稳定标识
  • elm：对应的真实 DOM 引用（patch 后才有）

关键点：VNode 不是为了“比 DOM 快”，而是为了：

  • 把渲染过程抽象成可比较的数据结构
  • 让更新变成：比较两棵树 → 做最少 DOM 操作

3.2 为什么不直接操作 DOM

DOM 操作的成本不只在“读写慢”，更在：

  • 触发布局/重排/重绘的链式代价
  • 操作不可控、难以批量合并
    VDOM 的价值：把多次状态变化合并成一次批量更新（watcher 队列 + nextTick），并且用 Diff 限制 DOM 改动范围。

4. 首次渲染：VNode 如何变成真实 DOM

首次渲染时，oldVnode 可能是一个真实 DOM 容器（挂载点），流程类似：

  1. 执行 render() 得到 vnode
  2. patch(container, vnode)
  3. patch 发现 old 是 DOM：走 createElm(vnode)
  4. 递归创建子节点、设置属性、绑定事件、插入 DOM
  5. 保存 vnode.elm 指向真实节点，后续更新复用

首次渲染没有 Diff 的“最小更新”意义，因为没有旧树可比，基本是“整棵树创建”。

5. 响应式如何驱动更新（更新为什么会发生）

5.1 Vue2：Object.defineProperty（你面试常说这一套）

  • 每个响应式属性都有 getter/setter
  • getter：当前正在渲染的 watcher（渲染 watcher）会被收集进 dep
  • setter：值变化 → dep.notify → watcher.update()

5.2 Vue3：Proxy（理解差异点即可）

  • Proxy 能拦截更多操作（新增/删除属性、数组索引等）
  • 依赖收集/触发机制仍是：track（收集） / trigger（触发）

5.3 watcher 队列与 nextTick：为什么不是立刻更新

数据可能在同一轮事件循环里被改很多次：

  • Vue 会把 watcher 去重入队
  • 在微任务/宏任务合适时机统一 flush
    效果：一次 UI 更新覆盖多次状态变动，降低抖动与重复渲染成本。

6. Diff：新旧 VNode 怎么比

Diff 的目标不是“算出所有不同”，而是在可接受复杂度内找到足够好的最小变更。Vue 的 Diff 基于几个关键假设：

  • 同级节点（siblings）之间的比较是主要场景（不做跨层级的最优匹配）
  • key 决定节点身份（尤其列表）
  • 同一个 vnode 是否“同一个节点”，取决于 sameVnode（Vue2）/ 等价判断（Vue3）

6.1 sameVnode（判断是否能复用真实节点）

典型判断维度：

  • tag 相同
  • key 相同（很重要）
  • 一些细节：是否是注释节点、输入类型等

如果不是 sameVnode：直接替换（删除旧的，创建新的）
如果是 sameVnode：复用 elm，进入精细对比（props/text/children）

7. Patch：Diff 之后怎么更新真实 DOM

Patch 做三类事：

  1. 更新节点自身：class/style/attrs/domProps/events
  2. 更新文本节点：text 不同就改 textContent
  3. 更新子节点 children：最复杂，也最核心

8. 核心：列表 children 的 Diff（双端比较）

Vue2 的经典策略是 双端指针 + key 映射（能背出来就很稳）：

设旧 children：oldStart ~ oldEnd
新 children：newStart ~ newEnd

循环比较四种命中（命中就移动指针并 patch）：

  1. oldStart vs newStart：头头相同 → patch，指针都 +1
  2. oldEnd vs newEnd：尾尾相同 → patch，指针都 -1
  3. oldStart vs newEnd：旧头 = 新尾 → patch，并把旧头对应 DOM 移动到末尾
  4. oldEnd vs newStart：旧尾 = 新头 → patch，并把旧尾对应 DOM 移动到开头

都不命中时：

  • 建一个 old 的 key -> index 映射表
  • 用 newStart.key 去找：
    • 找不到：说明是新增节点 → createElm 插入到合适位置
    • 找到了：取出旧 vnode → patch → 把它对应 DOM 移动到 newStart 前 → 旧位置标记为空

循环结束后：

  • 新的还有剩：批量新增
  • 旧的还有剩：批量删除

你要传达的重点：

  • Diff 主要优化的是同层列表重排
  • key 的存在让“查找复用节点”变成 O(1) 级别
  • 双端比较覆盖了大量头尾移动的高频场景

9. key 为什么这么重要

9.1 不加 key 的后果（或用 index 当 key）

Vue 可能采用“就地复用”策略：按位置对齐 patch
如果列表发生插入/删除/排序：

  • DOM 节点会被错误复用到别的数据项上
  • 表单输入、组件内部状态会串
  • 造成性能浪费（本该移动，结果变成大量 patch）

9.2 正确的 key

  • 应该是稳定且唯一：如 id、业务唯一标识
  • 不推荐用 index（除非列表不增删不重排、纯展示）

10. 从“改数据”到“页面更新”的口述闭环模板

  • 模板先被编译成 render 函数
  • 首次渲染执行 render 得到 VNode 树，patch 把它变成真实 DOM
  • 渲染过程中读取响应式数据会触发 getter，收集当前渲染 watcher
  • 当数据修改触发 setter，会通知 watcher 入队，nextTick 批量刷新
  • 刷新时重新执行 render 生成新 VNode，与旧 VNode 做 Diff
  • patch 把差异同步到真实 DOM，达到最小化更新
